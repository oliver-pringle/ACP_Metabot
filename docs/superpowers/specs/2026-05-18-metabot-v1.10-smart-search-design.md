## Summary

v1.10 layers **intent understanding** + **domain awareness** + **buyer-side
narration** on top of the retrieval engine v1.2/v1.7 already shipped. The
retrieval primitives (FTS5+dense+RRF+Voyage rerank) are solid; the gaps
now are *semantic* (DeFi-jargon expansion, intent classification),
*structural* (Resources not in unified search, no negative filters,
no sub-offering schema filters), and *narrative* (raw result lists vs
LLM-narrated explanations).

Seven changes, all backward-compatible:

1. **DeFi glossary expansion** in `SearchService` and `AgentSearchService` â€”
   ~200-entry static dictionary; expand query terms before retrieval.
2. **Resources in unified search** â€” `/v1/search` returns `offerings` + new
   `resources` array; relevant free Resources surface above paid offerings
   of similar relevance.
3. **Negative filters** on `/v1/search` and `/v1/searchAgents` â€”
   `excludeRequirements[]`, `excludeAgents[]`, `excludeChains[]`,
   `maxPriceUsd`.
4. **Sub-offering schema-aware filters** â€” index `requirementSchema` +
   `deliverableSchema` field names per offering; query can filter on
   `requiresField` / `producesField`.
5. **LLM query rewriter** (haiku tier, ~$0.0005/search) â€” gated by new
   `expand=true` flag on `/v1/search`; off by default for cost control.
   Adds 2-3 synonym queries, RRF-merges results.
6. **`searchNarrative` ($0.05 paid offering)** â€” Claude-narrated summary
   of top-5 search results with citations + 1-line "why this ranked high"
   per result. The AI-feel differentiator.
7. **`agentRiskCheck` ($0.05 paid offering)** â€” defensive scam-risk flag
   per agent (no reputation + outlier pricing + suspicious wallet pattern).
   Surfaces a `riskFlag` for buyers + can be hired standalone.

No retrieval-engine changes. Two new offerings (16 â†’ 18 paid). One new
Resource (`searchDebug` â€” free, exposes per-retriever raw scores for
transparency, 15 â†’ 16). ~20 new tests targeted at ~197 total (177 + 20).

## Why

- Metabot search precision is good for in-domain queries that share
  vocabulary with the offering descriptions. It collapses on **synonyms**
  ("HF" vs "health factor", "PoR" vs "proof of reserve") because the
  glossary lives only in buyer's head.
- The 15 free Resources are invisible in search. Buyers who'd benefit
  from `sourceCatalogue` (free) end up hiring `oracle_check` ($0.05)
  because they didn't know the Resource exists.
- Buyers can express positive intent ("LiquidGuard", "Aave") but not
  negative ("not LINK-dependent", "not Base-only", "under $0.20"). The
  data is all there; the API just doesn't accept it.
- The deliverable schema is a buyer's strongest signal about *what they
  get*. Today it's a free-text field; sub-offering filters turn it into
  facets.
- The "AI-feel" gap: result list vs narrated answer. Two buyers paste
  the same query, both get raw JSON â€” one buys, one bounces. A narrated
  summary closes the bounce rate.
- Scam risk is real and growing on V2 marketplace as registration
  volume increases. Surfacing a soft `riskFlag` is a defensive UX win
  AND a paid offering for risk-averse buyers.

## Scope and non-goals

In scope (this spec):

- Glossary file `Data/DeFiGlossary.json` + `QueryExpander.cs` service.
- `SearchService.SearchAsync` + `AgentSearchService.SearchAsync` call
  `QueryExpander.Expand(query)` before both legs of retrieval.
- `OfferingRepository.SearchResourcesAsync(...)` â€” new method; same
  hybrid pipeline shape but operating on the `resources` table.
- `/v1/search` response gains optional `resources` array.
- `SearchRequest` extended with `excludeRequirements`, `excludeAgents`,
  `excludeChains`, `maxPriceUsd`, `requiresField`, `producesField`,
  `expand` (bool).
- New `schema_facets` table: `(offering_id, field_name, role)` where role
  âˆˆ `{requirement, deliverable}`; populated by extraction pass on
  ingest.
- New `LlmQueryRewriter` service with `IQueryRewriter` abstraction â€”
  Anthropic Claude Haiku impl. Cost cap: $0.50/day across the bot
  (env `Search:QueryRewriterDailyUsdCap`); cap breach turns expansion
  off until next UTC midnight.
- New `SearchNarrator` service + `searchNarrative` offering.
- New `AgentRiskScorer` service + `agentRiskCheck` offering. Optional
  `includeRisk=true` on `/v1/search` and `/v1/searchAgents` surfaces
  a 1-byte `riskFlag` per hit at no charge.
- New free Resource `searchDebug?query=â€¦` returning per-leg scores +
  expansion list + reranker scores. Operator transparency, helpful for
  buyer LLMs that want to introspect.

Out of scope (parked for v2.0+):

- **HyDE** â€” needs a benchmark; defer until we see query types that
  actually fail under the existing hybrid+rerank.
- **Multi-vector indexing** â€” title/desc/req/del separate vectors. ROI
  uncertain on top of current single-vector + Voyage rerank.
- **Cross-encoder reranker upgrade** â€” Voyage is fine; swap to BGE-v2-m3
  only if we hit a cost ceiling on Voyage.
- **Buyer-intent classifier** â€” needs labelled training data we don't have.
- **`buyerPathfinder` (conversational multi-turn search)** â€” needs a
  separate conversation-state model; v2.0.
- **Composition recommendations on every search result** â€” extends
  `composeStack`; touches the orchestration model. v2.0.
- **Cross-chain awareness ranking** â€” needs an extraction pass on every
  offering's deliverable example. Useful but a small ship of its own;
  fold into v1.11 if v1.10 lands clean.
- **Learning-to-rank from `JobCreated` signals** â€” needs >1k labelled
  hires of training data. Capture the signal in v1.10 logs; train in v2.0.

## Architecture

Three architectural buckets:

### Bucket 1: Query enrichment (`QueryExpander` + `LlmQueryRewriter`)

```
buyer query
    â”‚
    â–¼
QueryExpander.Expand(q)
    â”‚   - glossary lookup: "HF" â†’ ["HF", "health factor"]
    â”‚   - lowercase + punctuation strip
    â”‚   - drop stopwords for FTS5 leg only
    â”‚   - if expand=true AND budget left: LlmQueryRewriter.RewriteAsync
    â”‚     â†’ ["original q", "synonym 1", "synonym 2"]
    â–¼
ExpandedQuery { primary: string, synonyms: string[], glossaryHits: string[] }
    â”‚
    â–¼
Both retrieval legs accept ExpandedQuery
    â”‚   - BM25 leg: OR all terms across primary + synonyms
    â”‚   - Dense leg: embed primary only; if rewriter active, embed
    â”‚     each synonym and average vectors (mean-pooling)
    â–¼
... existing RRF + rerank pipeline unchanged ...
```

- `Data/DeFiGlossary.json` ships in repo. ~200 entries to start; growable.
  Format: `[{ "canonical": "health_factor", "aliases": ["HF", "health
  factor", "loan-to-value", "LTV proxy"] }]`. Loaded once at boot into
  a `Dictionary<string, string[]>` keyed by lowercase alias â†’ canonical.
- `LlmQueryRewriter` uses Anthropic SDK with `claude-haiku-4-5-20251001`.
  Prompt template at `PromptTemplates/QueryRewrite.md`. Single LLM call,
  150-token max-output, JSON-mode for parsing. Cost ~$0.0005/call.
- Daily cap circuit-breaker: rolling 24h spend tracked in `query_rewrite_spend`
  table; when `Search:QueryRewriterDailyUsdCap` (default $0.50) is hit,
  `LlmQueryRewriter.RewriteAsync` returns the original query and logs a
  warning. Per-call cap also enforced (`Search:QueryRewriterMaxUsdPerCall`,
  default $0.002).

### Bucket 2: Search structure (`SearchService` extensions + Resources + schema facets)

- `SearchService.SearchAsync` signature gains:
  - `excludeRequirements: IReadOnlyList<string>?`
  - `excludeAgents: IReadOnlyList<string>?`
  - `excludeChains: IReadOnlyList<int>?`
  - `maxPriceUsd: decimal?`
  - `requiresField: string?`
  - `producesField: string?`
  - `includeResources: bool` (default true)
  - `expand: bool` (default false)
- Negative filters applied **after** retrieval+rerank but **before**
  pagination, so the top-N stays consistent with the buyer's filter intent.
- `OfferingRepository.SearchResourcesAsync(...)` runs the same hybrid
  pipeline against a new `resources_embeddings` table (mirrored from
  `agent_resources`). Re-uses Voyage rerank with the same query.
- Sub-offering filters (`requiresField`, `producesField`) hit the new
  `schema_facets` table (see Schema section).
- Response shape extension:
  ```jsonc
  {
    "offerings": [...],            // existing
    "resources": [...],            // NEW â€” free Resources matching query
    "expansion": {                 // NEW, omitted when expand=false
      "glossaryHits": ["HFâ†’health_factor"],
      "synonymQueries": ["liquidation warning system"],
      "rewriterCostUsd": 0.0005
    },
    "filtersApplied": {            // NEW â€” echoes what was applied
      "excludeRequirements": [],
      "maxPriceUsd": null,
      "requiresField": "tokenSymbol"
    }
  }
  ```

### Bucket 3: Narrator + Risk Scorer (new $0.05 offerings)

- **`SearchNarrator`**:
  - Inputs: query, top-5 hits (offering JSON), buyer's optional
    `previousQueries` array (for "I searched X earlier, now Y" continuity).
  - LLM: Claude Haiku, ~600 tokens out. Prompt template at
    `PromptTemplates/SearchNarrate.md`. Output: `summary` (3-5 sentences),
    `perResultReason` (array of 1-liners matching the top-5 ordering).
  - Cost: ~$0.0015/call against $0.05 price = healthy 30x margin.
  - Cached by `(query_canonical, corpus_version)` for 1 hour â€” repeat
    hires inside the cache window serve free of LLM cost.
- **`AgentRiskScorer`**:
  - Inputs: agent address, chainId.
  - Signals (each contributes 0-25 to a 0-100 risk score):
    1. **Reputation depth (0-25)** â€” `25 - min(25, jobCount Ã— 2)` from
       reputation history. New agents max out at 25.
    2. **Pricing outliers (0-25)** â€” average offering price vs category
       Ã— marketplace median. â‰¥3x median = 25; 1x median = 0.
    3. **Wallet provenance (0-25)** â€” does the wallet's funding history
       match known mixer/scam patterns? Uses a static
       `Data/SuspiciousFunderPatterns.json` (~50 entries: known mixer
       outputs, sanctioned addresses) â€” exact-match boolean check;
       not a heuristic, so no false-positive sprawl.
    4. **V1â†”V2 footprint anomaly (0-25)** â€” agents that exist in V2
       with identical-name offerings to a high-rep V1 agent score
       25 (likely impersonation). Looks up V1 by offering hash.
  - Returns `{ riskScore: 0-100, riskTier: "low|medium|high|critical",
    signals: [{name, score, detail}], lastEvaluatedAt }`.
  - Score thresholds: `low: 0-25`, `medium: 26-50`, `high: 51-75`,
    `critical: 76-100`.
  - Cached per agent for 6 hours.

## Schema (additions in `Db.cs`)

```sql
CREATE TABLE schema_facets (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    offering_id     INTEGER NOT NULL,
    field_name      TEXT NOT NULL,            -- 'tokenSymbol', 'txHash', 'chainId', etc.
    role            TEXT NOT NULL,            -- 'requirement' | 'deliverable'
    FOREIGN KEY (offering_id) REFERENCES offerings(id) ON DELETE CASCADE
);
CREATE INDEX ix_schema_facets_field ON schema_facets(field_name, role);
CREATE INDEX ix_schema_facets_off ON schema_facets(offering_id);

CREATE TABLE resources_embeddings (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    resource_id     INTEGER NOT NULL,
    embedding       BLOB NOT NULL,            -- Voyage embedding bytes
    embedded_at     TEXT NOT NULL,
    FOREIGN KEY (resource_id) REFERENCES agent_resources(id) ON DELETE CASCADE
);
CREATE INDEX ix_resources_embed_res ON resources_embeddings(resource_id);

CREATE VIRTUAL TABLE resources_fts USING fts5(
    resource_id UNINDEXED,
    name,
    description,
    tokenize = 'porter'
);

CREATE TABLE query_rewrite_spend (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    spent_at        TEXT NOT NULL,
    usd_amount      REAL NOT NULL,
    query_hash      TEXT NOT NULL,
    rewriter_model  TEXT NOT NULL
);
CREATE INDEX ix_rewrite_spend_at ON query_rewrite_spend(spent_at DESC);

CREATE TABLE search_narratives_cache (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    query_canonical TEXT NOT NULL,
    corpus_version  INTEGER NOT NULL,
    narrative_json  TEXT NOT NULL,
    generated_at    TEXT NOT NULL,
    UNIQUE(query_canonical, corpus_version)
);
CREATE INDEX ix_narr_cache_gen ON search_narratives_cache(generated_at DESC);

CREATE TABLE agent_risk_cache (
    agent_address   TEXT NOT NULL,
    chain_id        INTEGER NOT NULL,
    risk_score      INTEGER NOT NULL,
    risk_tier       TEXT NOT NULL,
    signals_json    TEXT NOT NULL,
    evaluated_at    TEXT NOT NULL,
    PRIMARY KEY (agent_address, chain_id)
);
```

Schema-facet population: when `OfferingRepository.UpsertAsync` runs, the
ingest path extracts top-level keys from `requirementSchema.properties`
and `deliverableSchema.properties` and writes them into `schema_facets`.
One-time backfill script `Maintenance/BackfillSchemaFacets.cs` runs once
on v1.10 boot if the table is empty.

Resources mirror: new `ResourcesEmbeddingsWorker` (runs nightly at
03:00 UTC, gated on `Resources:EmbedWorker:Enabled`) generates Voyage
embeddings for any `agent_resources` row missing an
`resources_embeddings` entry. Lazy backfill â€” won't run on a cold
deploy unless turned on.
